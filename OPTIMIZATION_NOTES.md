# 系統優化說明

## 優化日期
2026-03-22

## 優化內容

### 1. 前端 UX 優化

#### 問題 1：上傳按鈕卡住
**現象**：上傳大文件時，按鈕顯示"上傳中"超過 5 分鐘，沒有任何反饋

**解決方案**：
- 添加 60 秒上傳超時機制
- 超時後顯示友好的錯誤提示
- 允許用戶重新嘗試

**代碼位置**：`static/js/app.js` - `uploadFile()` 函數

#### 問題 2：進度顯示卡在 0%
**現象**：任務在排隊時，進度條顯示 0%，用戶以為系統卡住了

**解決方案**：
- 檢測 `queued` 狀態
- 顯示 "⏳ 排隊中，等待處理..." 提示
- 讓用戶知道系統正在工作

**代碼位置**：`static/js/app.js` - `checkStatus()` 函數

**建議進一步優化**：
- 顯示排隊位置（例如："排隊中，前面還有 1 個任務"）
- 顯示預估等待時間

#### 問題 3：長視頻拖動進度條卡住
**現象**：1 小時以上的視頻，拖動進度條時顯示 "video loading stopped"，需要滾動頁面才能恢復

**解決方案**：
- 添加 `preload="metadata"` 屬性，優化視頻加載策略
- 添加錯誤處理和自動重試機制
- 監聽 `seeking` 和 `seeked` 事件，提供更好的反饋
- 當發生解碼錯誤時自動重新加載視頻

**代碼位置**：
- `static/index.html` - video 標籤添加 `preload="metadata"`
- `static/js/app.js` - `setupEventListeners()` 中添加錯誤處理

**技術說明**：
- `preload="metadata"`：只預加載元數據（時長、尺寸等），不預加載整個視頻
- 減少初始加載時間，同時保持快速響應
- 對長視頻特別有效

#### 問題 4：缺少播放速度控制
**現象**：用戶無法調整播放速度，長視頻觀看效率低

**解決方案**：
- 添加播放速度控制面板（0.5x - 3x）
- 9 個速度選項：0.5x, 0.75x, 1x, 1.25x, 1.5x, 1.75x, 2x, 2.5x, 3x
- 懸停顯示，不遮擋視頻內容
- 支持全屏模式
- 自動同步到畫中畫視頻

**代碼位置**：
- `static/index.html` - 添加速度控制按鈕
- `static/css/style.css` - 速度控制樣式
- `static/js/app.js` - `setPlaybackSpeed()` 函數

**UX 設計考量**：
- 位置：右上角，懸停顯示（不干擾觀看）
- 視覺：半透明黑色背景，毛玻璃效果
- 反饋：當前速度高亮顯示（藍色）
- 交互：點擊即切換，無需額外確認
- 全屏：自動適配全屏模式位置

### 2. Google Translate 防封號優化

#### 問題：高頻率請求導致翻譯失敗

> **注意：** v2.5.0 已重寫翻譯服務為批量合併翻譯（多句合併為單次 API 請求），大幅降低請求頻率。
> 以下為舊版的單句翻譯策略記錄，僅作參考。

**現象**：
- 大量 "No translation was found" 錯誤
- 可能觸發 Google 的速率限制
- 長視頻翻譯時失敗率高

#### 解決方案 1：模擬人類行為
**實施的策略**：
- **隨機延遲**：每次翻譯間隔 0.3-0.8 秒（隨機）
- **批次休息**：每 20 個片段後休息 2-3 秒
- **減少批次大小**：從 50 降到 20

**代碼位置**：`src/services/translation_service.py`

```python
MIN_DELAY = 0.3  # 最小延遲（秒）
MAX_DELAY = 0.8  # 最大延遲（秒）
BATCH_DELAY = 2.0  # 批次間延遲（秒）
BATCH_SIZE = 20  # 批次大小
```

#### 解決方案 2：智能錯誤處理
**實施的策略**：
- **識別無法翻譯的內容**：公式、代碼、專有名詞自動保留原文
- **速率限制檢測**：檢測到 429 錯誤時，使用更長的延遲（5-11 秒）
- **隨機化重試延遲**：避免規律性被檢測

**錯誤類型處理**：
1. `no translation found` → 保留原文（不重試）
2. `rate limit` / `429` → 長延遲重試（5-11 秒）
3. 其他錯誤 → 標準重試（指數退避 + 隨機）

### 3. 性能影響分析

#### 翻譯速度變化
**之前**：
- 批次大小：50
- 無延遲
- 1 小時視頻（~645 片段）：約 2-3 分鐘

**現在**：
- 批次大小：20
- 每片段延遲：0.3-0.8 秒
- 批次間延遲：2-3 秒
- 1 小時視頻（~645 片段）：約 8-12 分鐘

**權衡**：
- ✅ 大幅降低被封號風險
- ✅ 減少翻譯失敗率
- ✅ 提高翻譯成功率
- ⚠️ 翻譯時間增加 3-4 倍
- ⚠️ 但對用戶體驗影響小（轉錄階段仍是最耗時的）

#### 總體處理時間
1 小時視頻的完整流程：
- 轉錄：10-30 分鐘（取決於 CPU）
- 翻譯（2 種語言）：16-24 分鐘
- 生成字幕：< 1 分鐘
- **總計**：約 26-55 分鐘

### 4. 可調整參數

如果需要調整速度/安全性平衡，可以修改以下參數：

```python
# 在 src/services/translation_service.py 中

# 更快但風險更高
MIN_DELAY = 0.1
MAX_DELAY = 0.3
BATCH_DELAY = 1.0
BATCH_SIZE = 30

# 更慢但更安全
MIN_DELAY = 0.5
MAX_DELAY = 1.5
BATCH_DELAY = 3.0
BATCH_SIZE = 10
```

1. **進度細化**：
   - 顯示當前翻譯進度（例如："翻譯中 245/645"）
   - 顯示預估剩餘時間

2. **排隊系統改進**：
   - 顯示排隊位置

   - 支持多個翻譯服務（DeepL, Azure, AWS）

   - 緩存常見片段的翻譯
   - 減少重複翻譯

3. **並行翻譯**：
   - 多語言並行翻譯（需要更多 API 配額）

   - 使用 Opus-MT 或類似模型
   - 完全避免 API 限制

2. **智能批次調整**：
   - 根據失敗率動態調整延遲
   - 自適應速率控制
